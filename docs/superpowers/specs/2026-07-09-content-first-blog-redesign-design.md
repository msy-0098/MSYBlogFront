# 内容优先博客重构设计稿

## 1. 背景与结论

本次重构以 `需求文档.md` 为准，保留博客站点已有的内容系统、评论体系、访客登录能力和后台管理能力，同时去掉 `Element Plus`，改为纯 CSS 的自研界面体系。

用户已确认的关键方向如下：

- 以需求文档为范围基准，补齐缺失功能，不只做表面换皮。
- 前台优先，优先提升公网访客体验。
- 首页走编辑专栏式内容首页，而不是科技演示页。
- 首屏主打一篇精选文章，强调内容代表作。
- 去掉粒子背景，改为静态高级感视觉语言。
- 文章详情页使用正文 + 目录侧栏的阅读结构。
- 首页首屏之后先接最新文章流。
- 文章列表和首页内容卡片以大图封面卡片为主。
- 后台必须支持封面图上传与正文插图上传，正文插图需要能写入 Markdown。

## 2. 当前现状与问题

### 2.1 前台现状

- 首页仍带有较强的粒子与动效表达，整体更像技术展示页，不够像长期写作型博客。
- 构建产物偏重，公共入口包体过大，后台依赖和公共前台没有充分隔离。
- 文章列表、详情、分类、项目等页面已经具备基础功能，但视觉体系不统一，内容层级不够稳定。

### 2.2 后台现状

- 后台大量依赖 `Element Plus`、`@element-plus/icons-vue` 和其消息/弹窗能力。
- 文章编辑已有封面上传，但正文 Markdown 插图工作流还不完整。
- 管理端视觉为库组件默认风格，和前台内容品牌感割裂。

### 2.3 后端现状

- 需求文档定位为 `SQLite + 轻博客 + 2C2G`，但当前后端实现已经偏向 `PostgreSQL`。
- 评论、访客注册登录、管理后台等能力已经存在，说明站点方向不再是纯静态博客，而是轻量内容平台。
- 搜索、上传、评论、设置等基础接口已有雏形，但仍需按最终产品工作流完善细节。

## 3. 目标与非目标

### 3.1 本次目标

- 完成前台内容体验重构，使站点成为“内容作品集式的个人博客”。
- 去除 `Element Plus`，建立纯 CSS 的前后台统一设计基础。
- 补齐需求文档中的公开页面与后台管理能力。
- 建立顺手的文章发布工作流，支持封面图和正文插图上传。
- 保证 `2C2G / 40GB` 服务器上可稳定部署、维护和长期更新。

### 3.2 非目标

- 不扩展成复杂社区产品，不加入多角色后台、复杂审核流或富媒体资产中心。
- 不引入大型前端 UI 框架替代 `Element Plus`。
- 不为“视觉炫技”保留重型 WebGL 背景。
- 不在本轮中引入 Elasticsearch、对象存储、消息队列等超出轻博客定位的基础设施。

## 4. 产品方向

### 4.1 整体产品定位

站点定位为“编辑专栏式个人技术博客”。它不是极简纯文字站，也不是项目作品堆砌页，而是以文章内容为主角、项目经历为辅助、个人定位为信任背书的内容品牌站点。

### 4.2 首页策略

首页承担三件事：

- 用精选文章快速建立专业感。
- 用最新文章流承接继续阅读。
- 用分类、项目和关于信息说明写作主题与作者能力。

首页结构固定为：

1. 顶部导航
2. 精选文章 Hero
3. 最新文章流
4. 专题分类入口
5. 精选项目
6. 关于我摘要
7. 页脚

### 4.3 页面范围

公开页面包含：

- 首页 `/`
- 文章列表 `/posts`
- 文章详情 `/posts/:slug`
- 分类列表与分类文章页
- 标签列表与标签文章页
- 归档页
- 项目页
- 关于页
- 搜索页
- 404 页

后台页面包含：

- 登录页
- 仪表盘
- 文章管理
- 新建/编辑文章
- 分类管理
- 标签管理
- 项目管理
- 评论管理
- 站点设置

## 5. 视觉与交互设计

### 5.1 前台视觉语言

前台采用静态高级感路线：

- 大面积留白
- 低饱和浅色背景
- 细线分隔和轻纸感纹理
- 克制渐变
- 杂志式大图封面
- 强标题、短摘要、清晰时间信息

不再保留粒子、球形穹顶、显著视差等会干扰阅读或拉高性能成本的表达。

### 5.2 字体与排版

- 标题使用更有书卷感和专栏气质的中文字体搭配稳重的英文字体。
- 正文排版优先长文阅读舒适度，保证行高、段距、列表、代码块和引用块层级清晰。
- 卡片标题比摘要更强，摘要比标签和元信息更强。

### 5.3 动效策略

- 保留轻量进入动效、淡入和滚动揭示。
- 不使用持续占用性能的实时背景动画。
- 所有动效默认服从 `prefers-reduced-motion`。

### 5.4 组件策略

前后台统一使用纯 CSS 自研基础组件：

- 按钮
- 输入框
- 下拉选择
- 标签
- 卡片
- 空状态
- 消息提示
- 模态确认框
- 分页
- 表格/列表容器

组件视觉上共用设计 token，但前台更具品牌感，后台更偏工具性。

## 6. 前台信息架构

### 6.1 首页

首页首屏采用双栏或大主视觉结构：

- 主区为精选文章，包含封面、标题、摘要、发布时间、分类、主按钮。
- 辅区为作者定位短句、站点说明和次级入口。

精选文章数据来源采用明确配置，不依赖“最新一篇自动顶替”。正式方案如下：

- 在站点设置中新增 `featuredPostSlug` 字段。
- 若配置失效，则回退到最新一篇已发布文章。

### 6.2 最新文章流

- 首页精选区下方紧接最新文章流。
- 文章以大图封面卡片展示，但每张卡片仍以标题和摘要为主要信息。
- 首页默认展示最近 6 篇文章。

### 6.3 分类、项目、关于

- 分类区展示核心主题及文章数量。
- 项目区只展示精选项目，强调代表作而不是罗列全部。
- 关于区只放摘要和跳转，详细个人信息留到独立关于页。

### 6.4 文章详情页

详情页结构：

- 文章头部：封面、标题、摘要、发布日期、分类、标签
- 正文主栏：Markdown 内容
- 目录侧栏：桌面端常驻，移动端折叠
- 底部辅助区：上一篇/下一篇、相关文章、评论区

阅读体验要求：

- 正文宽度稳定，不因目录栏挤压得过窄。
- 图片懒加载，超宽图片不破版。
- 代码块支持横向滚动和高亮。
- 外链默认新窗口打开。

### 6.5 列表、分类、标签、搜索、归档

- 列表页统一成同一套卡片体系。
- 分类与标签页不仅提供筛选结果，也要提供上下文标题和数量信息。
- 搜索页支持空关键词提示、无结果状态、结果列表延续同一卡片风格。
- 归档页按年份与月份聚合，视觉上更像目录索引页而不是原始数据表。

## 7. 后台体验设计

### 7.1 总体策略

后台不追求花哨，重点是清晰、顺手和连续工作流。页面结构统一为：

- 顶部区域：标题、简介、主操作
- 主内容区：列表/表单
- 辅助反馈：提示条、确认弹窗、空状态、加载态

### 7.2 文章编辑工作流

文章编辑页分为两大区块：

- 内容区：标题、摘要、正文 Markdown 编辑
- 发布区：Slug、分类、标签、状态、发布时间、封面图

正文插图上传方案：

- 编辑器提供“上传图片”操作。
- 选图后调用后台上传接口。
- 上传成功后自动将 `![alt](url)` 插入到当前光标位置。
- 上传失败时给予明确错误提示。

### 7.3 上传能力

本轮支持：

- 文章封面图上传
- 项目封面图上传
- 正文插图上传

本轮不做完整媒体库，但接口和数据表继续保留上传记录，便于后续扩展。

### 7.4 评论与访客能力

- 保留访客注册、登录、评论能力。
- 评论提交流程保持轻量，前台以阅读体验为先，后台提供审核/隐藏/删除管理。
- 不扩展复杂通知或互动体系。

## 8. 技术方案

### 8.1 前端架构

前端按“公共站点”和“管理端”两条路径拆分：

- 公共站点只加载公共所需依赖。
- 管理端路由全部懒加载。
- 移除全局 `Element Plus` 注册与样式导入。
- 用自研样式系统替换现有 UI 库组件。

前端需要建立的基础能力：

- 设计 token：颜色、间距、圆角、边框、阴影、字号
- CSS 分层：基础、布局、组件、页面
- 统一交互反馈：toast、modal、inline error
- 管理端基础控件：table、pagination、input、select、tabs、confirm

### 8.2 性能收束

性能目标：

- 首页不再依赖 WebGL 背景。
- 管理端资源不进入公共首页首屏。
- Markdown 渲染和代码高亮尽量只在详情页相关 chunk 中加载。
- 图片资源懒加载，封面图控制尺寸与裁剪策略。

### 8.3 后端架构

后端继续保持单体轻服务模式：

- `Gin` 负责 HTTP 路由
- `GORM` 负责数据访问
- `JWT` 负责鉴权
- 上传文件走本地磁盘目录

数据库方向回归需求文档，使用 `SQLite` 作为本轮唯一正式支持方案。原因如下：

- 更贴合 `2C2G` 服务器规模
- 运维复杂度更低
- 与项目的个人博客定位一致
- 当前数据量与访问模型完全足够支撑

本轮不保留多数据库并行支持目标，后端配置、迁移、测试与部署全部以 `SQLite` 为准。

### 8.4 搜索策略

搜索正式采用 `SQLite FTS5`，对标题、摘要和正文建立统一全文索引，不再继续把简单 `LIKE` 查询作为最终方案，也不引入独立搜索中间件。

## 9. 数据与接口调整

### 9.1 站点设置

站点设置新增或明确以下内容：

- `featuredPostSlug`
- 首页作者简介/定位文案
- 社交链接
- 关于页展示文本

### 9.2 上传接口

上传接口要求：

- 登录后可调用
- 限制图片类型与大小
- 失败返回明确错误原因
- 返回可直接写入 Markdown 的访问路径

上传目录结构固定为：

- `/uploads/posts/yyyy/mm/...`
- `/uploads/projects/yyyy/mm/...`
- `/uploads/content/yyyy/mm/...`

### 9.3 评论接口

评论接口保留当前访客鉴权模型，但前台加载采用异步策略：

- 文章正文先渲染
- 评论区延后加载
- 评论失败不影响正文浏览

## 10. 错误处理与边界场景

必须明确处理以下场景：

- 精选文章配置失效
- 文章封面缺失
- 上传图片格式不合法或超限
- Slug 冲突
- 分类删除时仍有关联文章
- 访客未登录评论
- 搜索空结果
- 文章不存在或被隐藏
- `/uploads` 资源丢失

所有场景都需要前端可理解提示，不允许只有控制台报错。

## 11. 测试与验收

### 11.1 前台验收

- 首页结构符合“精选文章 -> 最新文章 -> 分类 -> 项目 -> 关于”的节奏
- 首页无粒子背景，整体气质稳定、安静、可读
- 列表页、详情页、分类/标签/搜索/归档视觉统一
- 详情页目录在桌面端与移动端都可用

### 11.2 后台验收

- 登录、鉴权、跳转正常
- 文章创建、编辑、删除、发布状态切换正常
- 分类、标签、项目、设置、评论管理完整可用
- 封面上传和正文插图上传工作流顺畅

### 11.3 技术验收

- 前端能通过 `pnpm build` 与测试
- 后端能通过构建与关键测试
- 前后台资源分离明显，公共前台包体显著下降
- 生产部署路径继续兼容 `Nginx + systemd`

## 12. 分阶段实施建议

### 阶段 1：建立设计系统与公共骨架

- 移除公共入口对 `Element Plus` 的依赖
- 建立 token 与基础组件
- 重做导航、页脚、公共容器和页面节奏

### 阶段 2：完成前台核心页面

- 首页
- 文章列表
- 文章详情
- 分类、标签、归档、搜索
- 项目页与关于页

### 阶段 3：补齐内容运营工作流

- 精选文章配置
- 封面策略
- 评论区体验
- 404 与空状态

### 阶段 4：后台脱离 UI 库

- 登录
- 仪表盘
- 列表页
- 编辑页
- 设置页
- 提示与确认交互

### 阶段 5：后端收束与上线验证

- 恢复 `SQLite` 默认路径
- 校正上传、评论、设置、搜索等关键链路
- 做部署与性能回归

## 13. 风险与缓解

- 风险：前后台同时改动过多，导致节奏失控。
  - 缓解：按阶段交付，先前台后后台。
- 风险：去掉 `Element Plus` 后组件回归工作量偏大。
  - 缓解：先做组件底座，再替换页面。
- 风险：数据库从 `PostgreSQL` 回到 `SQLite` 可能涉及适配差异。
  - 缓解：优先梳理驱动差异与迁移脚本，再做业务回归测试。
- 风险：大图封面卡片对上传质量依赖高。
  - 缓解：后台加入尺寸建议、比例提示和默认封面策略。

## 14. 最终结论

本次重构不是简单换皮，而是一次围绕“内容优先、前台优先、轻量稳定、长期维护”的整体收束。最终产品应表现为：

- 前台像一份有品牌感的技术专栏
- 后台像一套顺手的内容发布工具
- 后端像一套适合个人博客长期运行的轻量服务

在这一前提下，最推荐的实施路线是：

1. 先完成前台设计系统与内容体验重构
2. 再补顺图片上传与内容管理工作流
3. 最后彻底去掉 `Element Plus` 并收束后端到轻博客最佳形态
